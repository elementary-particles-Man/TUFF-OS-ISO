# TUFF-FS 空き容量管理（アロケータ）詳細仕様書

## 1. 概要
TUFF-FS における空き容量管理は、物理ブロックの利用状況を管理する「Bitmap」と、トランザクション整合性を担保するための「Epoch-aware Reclaim（エポック連動型再利用）」の二層構造で実装されている。

## 2. 物理ブロックビットマップ (Physical Bitmap)
ストレージ全体の 4KB ブロックに対して 1ビットを割り当て、使用状況を管理する。

- **配置**: `Genesis` ブロックによって定義された「Bitmap Zone」に配置。
- **読み込み**: 起動時に ZRAM 上へキャッシュされ、高速なビット演算によるアロケーションを行う。
- **3N冗長化**: 3枚の物理ディスクに対して独立したビットマップが存在し、`MetadataCriticalPlacementPolicy` に基づき、複製ごとに異なる物理 LBA が選択される。

## 3. アロケーション・ライフサイクル

### 3.1. ブロック予約 (Reservation)
書き込みが発生した際、アロケータはビットマップから空きブロックを探し、「予約（Reserved）」マークを付ける。
この時点ではビットマップの永続化層（物理ディスク）は更新されず、UQ（Unique Queue）内のメモリ上で管理される。

### 3.2. 確定 (Commit)
`fs commit` 実行（Epoch フリップ）により、予約されたブロックが正式に「使用中（Allocated）」としてビットマップへ永続化される。

### 3.3. 遅延再利用 (Deferred Reclaim)
TUFF-FS の最も特徴的な機能である。データの削除や更新により不要になったブロックは、**直ちに空きブロック（Free）には戻されない。**

- **ReclaimPendingEntry**: 不要になったブロックは、このリストに「どの Epoch で不要になったか」と共にキューイングされる。
- **再利用の条件**: J世代（ジャーナリング）の整合性を守るため、設定された保持期間（`MqRetentionPolicy`）および Epoch 数が経過するまで、ブロックは予約状態（再利用不可）として保護される。
- **ReaderTracking**: `ActiveChunkRead` によって現在進行中のリード処理がある場合、その処理が完了するまで再利用はさらに延期される。

## 4. 主要な構造体（tuff-core/src/fs_structs.rs）

| 構造体 | 役割 |
|:---|:---|
| `ReclaimState` | 再利用待ちリスト（pending_list）と即時利用可能リスト（free_list）を管理。 |
| `ReclaimPendingEntry` | ブロックID、破棄されたEpoch、再利用解禁Epochを保持。 |
| `ReaderReclaimPolicy` | アクティブな読み込みセッションがある場合の保護ポリシーを定義。 |
| `MetadataCriticalPlacementPolicy` | 3N冗長化時における、物理セクタのストライド（間隔）やスロットを決定。 |

## 5. 障害耐性
- **Bitmap Inconsistency**: `fs fsck` 実行時、ビットマップと実際の FMC (File Metadata Chunk) のポインタをクロスチェックし、浮いているブロックや二重割り当てを自動修復する。
- **Isolation Mode**: システムが隔離モード（Isolation）に移行した場合、アロケータはすべての新規予約を拒否し、ビットマップの更新を物理的にロックする。

## 6. 結論
TUFF-FS のアロケータは単なる「空き領域探し」ではなく、**「J世代の歴史を物理的に保護しながら、安全に再利用サイクルを回す」**ためのインテリジェントな時間軸管理機構として機能している。
